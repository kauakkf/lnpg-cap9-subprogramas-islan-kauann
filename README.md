# Discussão

Nesse programa de Sistema Bancário foram usados os conceitos de Tipos Abstratos de Dados, encapsulamento, abstração e subprogramas, que vimos anteriormente nas aulas. O projeto ficou dividido em cinco módulos principas: o Cliente, o Banco, a ContaBancaria, o Validador e o Main.
A classe Validador foi criada para que não houvesse repetições durante todo o código. Ela fica responsável por condensar as regras da validação dos dados digitados. Nela temos os métodos nome_valido, cpf_valido, telefone_valido e email_valido. Assim, é possível que essas regras sejam reutilizadas em partes diferentes do programa, o que torna o código mais limpo e fácil de manter. Se por acaso uma regra precisar ser alterada, isso acontece apenas uma vez, no Validador.
Os relacionamentos das entidades foram implementadas pela associação entre os objetos. A classe ContaBancaria tem referências para o objeto do Cliente, que representa o titular da conta, e para o objeto do Banco, que representa a instituição financeira responsável por essa conta. A classe banco tem uma lista de todas as contas cadastradas, de forma que, um banco pode reunir várias contas e um cliente também pode estar associado a mais de uma conta.
As regras de validação são um ponto de atenção maior, porque garante que o usuário forneça os dados corretamente e evita duplicação, por exemplo. Por isso os nomes precisam de um tamanho mínimo, o CPF obrigatoriamente tem 11 dígitos, além do telefone com 10 ou 11 dígitos e o e-mail que precisa conter um "@". A classe Banco não permite cadastrar duas contas com o mesmo número.
Os conceitos de encapsulamento foram usados ao manter os dados e os comportamentos organizados cada um na sua classe, onde cada uma delas tem responsabilidades específicas e oferece os métodos para a manipulação dos dados. Essa divisão das responsabilidades de cada arquivo facilita para otimizar a legibilidade e a manutenção do programa, além de permitir expansões futuras com mais tranquilidade.
